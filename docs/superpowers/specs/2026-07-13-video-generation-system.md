# Wolves Soundtrack Video Generation System

## Goal

Every song added to the Wolves YouTube playlist (`PLA78oiE-RGAE`) should
automatically get a companion "lore video" generated by the project — a
1080p/60 MP4 that pairs the track's audio with a beat-synced slideshow of
Wolves wallpapers, artwork, and lore quote overlays. The output is intended
for uploading as an unlisted YouTube video, embedding on the Wolves page as a
fallback when the YouTube IFrame is blocked, and as a downloadable artifact
for the community.

The critical constraint from the operator: **this is a pipeline, not a
one-off**. Adding a new track to the playlist must produce a new video with
zero manual editing. Adding a new wallpaper or lore entry must automatically
enrich all future renders. Re-rendering an existing track (because lore
changed, wallpapers changed, or the template improved) must be a single
command against a stable, content-addressed cache.

## Non-Goals

- No hand-authored per-song timelines. Every render is data-driven.
- No proprietary NLE (Premiere, Resolve, Final Cut). The pipeline must run
  headless in GitHub Actions on Ubuntu runners.
- No re-encoding of the YouTube stream itself. Audio is fetched via `yt-dlp`
  at pipeline time from the playlist entries already tracked in
  `wolves-playlist.json`.
- No live video effects (particle systems, 3D). Ken Burns + crossfades +
  typographic overlays only. The aesthetic matches the site.
- No shipping the generated MP4s in the git repo. They are release
  artifacts / uploaded to YouTube, not committed.

## Design Principles

1.  **Data in, video out.** The renderer is a pure function of
    `(track metadata, wallpaper list, lore entries, template version)`. Same
    inputs, same output, byte-for-byte where possible.
2.  **Reuse the source of truth.** The site already owns the playlist
    (`public/wolves-playlist.json`), wallpaper index
    (`src/components/wolves/wallpapers-list.ts`), and lore
    (`src/components/wolves/lore.ts`). The video system consumes these; it
    does not fork them.
3.  **Beat-aware, not clock-aware.** Slide intervals already encoded in the
    playlist (`bpm`, `phraseBeats`, `slideInterval`, `fadeDuration`) drive
    cuts. New tracks inherit safe defaults until tempo metadata is added.
4.  **Content-addressed cache.** Renders are cached by a hash of their inputs
    so CI never re-encodes work it has already done. Only new/changed tracks
    cost minutes.
5.  **Progressive enrichment.** A track with only a title and artwork still
    renders (fallback template). Adding tempo metadata or a matched lore
    chapter unlocks richer output without changing the pipeline.
6.  **One template, many songs.** All artistic decisions live in a single
    Remotion composition. Editing the template re-skins every video in the
    catalog on the next render.

## Tech Stack

| Layer | Choice | Rationale |
|---|---|---|
| Composition engine | **Remotion 4** (React) | React-based, deterministic, first-class GitHub Actions rendering, native audio+video timeline, ships its own headless Chromium. The site is already Vue/TS; Remotion's React island is scoped to `video/` and does not touch the app bundle. |
| Encoder | **FFmpeg** (Remotion's bundled build) | Standard, well-known flags, hardware-agnostic. Also used directly for the audio-only prep step and the final loudness normalization pass. |
| Audio acquisition | **yt-dlp** (already a repo dependency via `update-wolves-playlist.js`) | Same tool the playlist updater uses. Extracts best-quality audio (`bestaudio[ext=m4a]`) per YouTube video id. |
| Audio normalization | **ffmpeg -af loudnorm** (EBU R128, `-16 LUFS`, `-1.5 dBTP`) | Consistent perceived loudness across an artist-diverse playlist. |
| Waveform / beat data (future) | **aubio** or **essentia** | Optional. Only invoked when a track lacks `bpm` in the playlist; falls back to slide-interval defaults. |
| Subtitles / captions | **SRT** generated from lore entries and dropped as a sidecar for YouTube auto-upload | Overlays are burned in via Remotion; SRT is exported for YouTube's caption track. |
| Orchestration | **GitHub Actions**, extending `.github/workflows/update-content.yml` and a new `render-wolves-videos.yml` | Same runner model as the rest of the pipeline. Daily cron + `workflow_dispatch`. |
| Cache | **GitHub Actions cache** keyed on the input hash, plus optional S3-compatible bucket (Backblaze B2 / R2) for the encoded MP4 artifacts | Actions cache handles the fast path; object storage handles the durable, addressable output. |
| Upload (optional, later phase) | **YouTube Data API v3** via `google-auth-library` in a small Node script | Only enabled once the operator provisions the credentials; the render pipeline works standalone without it. |

Remotion is chosen over Melt/MLT and raw FFmpeg filtergraphs because:

- Melt/MLT XML is powerful but hostile to code review; per-song timelines
  become unreadable fast and diffing renders is painful.
- Raw FFmpeg `-filter_complex` graphs collapse under Ken Burns +
  typography + per-track tempo. Debugging is print-driven.
- Remotion lets us write **one React component** whose props are the track,
  and get back a deterministic timeline. The overlap with our existing TS/JSX
  skill set is total.

## Data Model

### `VideoTrackProps` (fed to the Remotion composition)

```ts
interface VideoTrackProps {
  track: {
    id: string
    title: string
    artist: string
    youtubeVideoId: string
    artworkUrl: string        // resolved absolute path or file:// URL
    audioPath: string         // local m4a from step 2
    durationSec: number       // probed with ffprobe
    bpm?: number
    phraseBeats?: number
    slideIntervalSec?: number // fallback to 8s
    fadeDurationMs?: number   // fallback to 1200
  }
  chapter?: {
    id: string
    title: string
    palette: 'day' | 'night' | 'duality'
  }
  wallpapers: Array<{ path: string, title: string }>  // pre-picked set
  lore: Array<
    | { kind: 'quote', text: string, attribution: string }
    | { kind: 'conversation', title: string, lines: string[] }
  >
  templateVersion: string     // e.g. "v1.2.0", bumped on template edits
}
```

### Input hash (cache key)

```
sha256(
  JSON.stringify({ track, chapter, wallpapers, lore, templateVersion })
)
```

The hash is what lives in `dist/videos/<hash>/<id>.mp4` and what the CI
cache is keyed on. A new lore entry changes the hash for every track in
that chapter and triggers re-render only for the affected songs.

## Pipeline Flow

### Local developer flow

```bash
npm run video:manifest        # step 1, deterministic, no network
npm run video:fetch-audio     # step 2, uses yt-dlp; skips cached files
npm run video:render -- --id LASru9j0oIc     # single track
npm run video:render                          # entire playlist
npm run video:preview -- --id LASru9j0oIc    # opens Remotion Studio
```

`video:preview` is the killer feature: Remotion Studio gives a live,
scrubbable timeline against real audio while iterating on the template.

### CI flow (`.github/workflows/render-wolves-videos.yml`)

Triggers:

- `workflow_dispatch` (manual, with optional single-track input)
- After `update-content.yml` runs (via `workflow_run`), so new playlist
  entries are picked up the next day.
- On `push` to `main` that touches `video/**`, `src/components/wolves/lore.ts`,
  or `src/components/wolves/wallpapers-list.ts`.

Job graph:

1.  **plan** — checkout, `npm ci`, run `build-video-manifest.js`, emit a
    matrix of `{ trackId, inputHash }`. Filter out any hash already present
    in the object-storage bucket.
2.  **render** — matrix over the plan output, one runner per track.
    - Restore `~/.cache/yt-dlp` and `tmp/audio/` from Actions cache keyed on
      `youtubeVideoId`.
    - `fetch-track-audio.js` (only downloads if missing).
    - `render-wolves-videos.js` (invokes Remotion; concurrency=1 per runner,
      GPU-less, ~4-8 min per song on `ubuntu-latest`).
    - Upload MP4 + SRT to bucket, keyed by `<hash>/<trackId>.mp4`.
3.  **index** — after `render` fan-in, regenerate `public/wolves-videos.json`
    from bucket contents and open a PR (same pattern used by
    `update-content.yml`) so the site gains the new video links on merge.
4.  **(optional) publish-youtube** — gated on
    `secrets.YOUTUBE_REFRESH_TOKEN`; uploads any track that does not yet
    have a `youtubeGeneratedVideoId` in the index.

## Integration Points

### `scripts/update-wolves-playlist.js`

No behavioural change required. It continues to be the sole writer of
`public/wolves-playlist.json`. The video manifest step reads that file
downstream. This keeps concerns separated: playlist ingestion is one
job; video rendering is another.

Optional enhancement (later): after the playlist updater discovers a new
track, dispatch `render-wolves-videos.yml` via `gh workflow run` so the
new song's video is ready within the same day rather than the next cron.

### `WolvesSoundtrack.vue`

Adds a `fallbackVideoUrl` field per track, sourced from
`public/wolves-videos.json`, used when the YouTube IFrame API errors
(country blocks, ad blockers, etc.). The current component's state
machine already has an `error` state; wiring a `<video>` element into
that branch is a small follow-up, not part of this system's scope.

### Object storage

Two acceptable backends: **Cloudflare R2** or **Backblaze B2**. Both are
S3-compatible and cheap for infrequent-access media. Bucket layout:

```
wolves-videos/
  by-hash/<hash>/<trackId>.mp4
  by-hash/<hash>/<trackId>.srt
  latest/<trackId>.mp4              # symlink-style redirect to newest hash
```

`by-hash/` is immutable; `latest/` is what the site links to.

## Template Behaviour (one composition, many songs)

`WolvesSongVideo` renders, in order:

1.  **0.0s – 2.5s: TitleCard.** Fade-in of title/artist over a blurred
    version of the artwork, Bluefin ocean gradient underlay. Bottom-right
    corner shows the chapter badge if present.
2.  **2.5s – end-4.0s: SlideshowScene.**
    - Wallpapers cross-fade every `slideIntervalSec` (fallback 8s), Ken
      Burns pan/zoom over `slideIntervalSec + fadeDurationMs`.
    - Lore overlays appear on phrase boundaries derived from `bpm` and
      `phraseBeats`. Quotes fade in bottom-left in JetBrains Mono over a
      subtle scrim; conversations render as a "transcript" stack.
    - Wallpaper selection is deterministic: chapter palette + track id
      as seed against the wallpaper index, so a given track always picks
      the same set.
3.  **end-4.0s – end: OutroCard.** QR to
    `projectbluefin.io/wolves#<chapterId>` and the track's YouTube link,
    with `Assisted-by: <templateVersion>` in small text bottom-right for
    traceability.

Audio: the fetched `.m4a` is placed on the timeline via Remotion's
`<Audio />`. Video duration = audio duration. No cropping; the outro
overlaps the last four seconds of audio via opacity.

## Failure Modes and Recovery

- **yt-dlp breaks on a video** (private/removed): manifest builder skips
  the track and writes a warning row into `public/wolves-videos.json`
  under `errors[]`. Site can surface a "video unavailable" state.
- **Remotion render OOM** on a runner: retry once on `ubuntu-latest-4-core`
  (larger runner). Failing that, the job is marked failed for that single
  track; the rest of the matrix continues.
- **Template regression** (bad edit in `video/`): CI runs `render` for a
  single known-good track (`LASru9j0oIc`, 7 Days to the Wolves) as a smoke
  test before fanning out. If it fails, no other renders start.
- **Bucket upload fails**: MP4 is uploaded as a GitHub Actions artifact
  as a fallback so nothing is lost, and the index step is skipped.
- **Cache poisoning**: the input hash covers everything the template
  reads. A `templateVersion` bump invalidates all hashes on purpose.

## Rollout Plan

- **Phase 0 — Scaffold.** Land the `video/` Remotion project, the four
  scripts, and the workflow file. Render one track (7 Days to the
  Wolves) end-to-end locally. No CI enabled yet.
- **Phase 1 — CI-driven catalog render.** Enable the workflow on manual
  dispatch only. Render the entire current 7-track playlist to R2/B2.
  Hand-review the outputs. Iterate on the template.
- **Phase 2 — Auto-refresh.** Chain onto `update-content.yml` so new
  playlist entries produce videos within a day. Land
  `public/wolves-videos.json` and wire the fallback player.
- **Phase 3 — YouTube upload.** Add the optional publish step behind a
  secret. Generated videos land as unlisted uploads on the Bluefin
  channel, cross-linked from the Wolves page.
- **Phase 4 — Beat detection.** For tracks lacking `bpm` in the playlist,
  run `aubio tempo` during audio fetch and write the result back into
  `wolves-playlist.json` via the same PR flow used by
  `update-content.yml`.

## Success Criteria

- A brand-new track added to playlist `PLA78oiE-RGAE` results in a
  rendered MP4 in the bucket within 24 hours, with zero manual work.
- Re-rendering the full 7-track catalog after a template edit costs one
  `render-wolves-videos.yml` run and produces 7 new MP4s with new
  hashes, while the old hashes remain retrievable.
- The Wolves page gracefully falls back to the generated video when the
  YouTube IFrame errors.
- The pipeline runs entirely on GitHub-hosted runners and free-tier
  object storage; no local render farm required.
